# Database Design Guide

## Overview

The SQLite database is the kiosk's source of truth. It stores student records, cards, authentication credentials, audit logs, and batch metadata.

---

## Key Design Decision 1: Email + SMS Credential Delivery

### Purpose

The `email` field in the `students` table enables **dual-channel credential delivery**. OTP and temporary PINs are sent to both the student's registered email and phone number for reliability and accessibility.

**Advantages:**

- **Redundancy** ŌĆö If SMS fails (network, provider issues), email delivery succeeds
- **Accessibility** ŌĆö Some students may not have active SMS receive (international numbers, spotty coverage)
- **Industry Standard** ŌĆö Multi-channel 2FA is the modern security best practice

### Implementation

Each student record has:

- `email` ŌĆö University email (TEXT, can be NULL for edge cases)
- `phone_number` ŌĆö Mobile phone for SMS (TEXT, can be NULL)

When credentials are generated:

```bash
send_credentials(email, phone_number, otp, temp_pin=None):
  - Send SMS via Africa's Talking if phone_number exists
  - Send email via SMTP (Gmail/University SMTP) if email exists
  - Log failures separately for each channel
  - Do NOT fail the entire batch if one channel fails
```

**Message Examples:**

For **Returning Students** (no temp PIN):

- **SMS:** `"Your ID card is ready. OTP: 847291. Collect 24/7."`
- **Email:** `Subject: Your ID Card Claim OTP`  
  `Your OTP is: 847291 (valid 24 hours)`

For **First-Year Students** (with temp PIN):

- **SMS:** `"OTP: 847291 | Temp PIN: 3829. Change PIN on first visit."`
- **Email:** `Subject: Your ID Card Claim - OTP and Temporary PIN`  
  `OTP: 847291 | Temporary PIN: 3829 (Change on first collection)`

## Key Design Decision 2: `is_temp_pin` Column

### Purpose

The `is_temp_pin` column in the `authentication` table distinguishes between:

- **Temporary PINs** (`is_temp_pin = TRUE`) ŌĆö System-generated 4-digit codes given to **first-year students** on their first card collection
- **Permanent PINs** (`is_temp_pin = FALSE`) ŌĆö User-chosen codes set by students after first collection or used by **returning students** who already have a PIN

### Workflow

#### First-Year Student (New to System)

1. **Card Loaded into Kiosk** ŌĆö OCR reads their registration number during staff batch loading
2. **SMS Sent** ŌĆö `send_otp_sms()` generates both:
   - A 6-digit OTP (random, cryptographically secure)
   - A 4-digit temporary PIN (random, generated by `generate_temp_pin()`)
   - Both are hashed before storage
3. **Database State After SMS**

   ```bash
   authentication table:
   - otp_hash = bcrypt(otp)
   - otp_expiry = now + 24 hours
   - pin_hash = bcrypt(temp_pin)
   - is_temp_pin = TRUE  ŌåÉ Marks this as temporary
   - failed_otp_attempts = 0
   - failed_pin_attempts = 0
   ```

4. **Student Enters OTP** ŌåÆ Successfully verified via `verify_otp()`
5. **Student Enters Temp PIN** ŌåÆ Successfully verified via `verify_pin()`
6. **Check `is_temp_pin` Flag** ŌåÆ If TRUE:
   - Force the **PIN Setup Screen** ŌåÆ Student enters a new permanent PIN twice to confirm
7. **Set Permanent PIN**

   ```sql
   UPDATE authentication
   SET pin_hash = bcrypt(new_pin), is_temp_pin = FALSE
   WHERE registration_number = ?
   ```

8. **Confirm Collection** ŌåÆ Card dispensed, session ends

#### Returning Student (Has Prior PIN)

1. **Card Scanned via Expired Card Slot** ŌåÆ OCR reads their old card
2. **Database Already Has Record** ŌåÆ From previous year, `is_temp_pin = FALSE` (permanent PIN)
3. **OTP Generated and Sent**

   ```bash
   authentication table:
   - otp_hash = bcrypt(otp)
   - otp_expiry = now + 24 hours
   - pin_hash = [unchanged ŌĆö still their permanent PIN]
   - is_temp_pin = FALSE  ŌåÉ Still permanent, not temp
   ```

4. **Student Enters OTP** ŌåÆ Verified
5. **Student Enters PIN** ŌåÆ Verified against their existing PIN hash
6. **Check `is_temp_pin` Flag** ŌåÆ If FALSE:
   - Skip PIN setup screen, proceed directly to **Confirmation Screen**
7. **Confirm Collection** ŌåÆ Card dispensed

## Implementation Notes

### During Authentication

**In `auth.py` or equivalent:**

```python
def verify_pin_and_check_temp(reg_number, submitted_pin):
    # Verify PIN against hash
    is_valid = verify_pin(reg_number, submitted_pin)

    if not is_valid:
        # Handle failure, increment counter, check lockout
        return False

    # PIN was correct ŌĆö now check if it's temporary
    is_temp = get_is_temp_pin_flag(reg_number)

    if is_temp:
        # Force permanent PIN setup before collection
        return REQUIRES_PIN_SETUP
    else:
        # Proceed to collection confirmation
        return SUCCESS
```

### During Permanent PIN Setup

**In `auth.py` or equivalent:**

```python
def set_permanent_pin(reg_number, new_pin):
    # Hash the new PIN
    pin_hash = bcrypt.hashpw(new_pin.encode(), bcrypt.gensalt())

    # Update database
    DB.execute(
        "UPDATE authentication SET pin_hash = ?, is_temp_pin = FALSE WHERE registration_number = ?",
        (pin_hash, reg_number)
    )
```

### Database Cleanup (Phase 2, Task 2.5)

When deleting old collected card records older than 120 days, also delete their associated authentication rows:

```python
def cleanup_old_records():
    # Delete authentication records for collected cards older than 120 days
    DB.execute(
        """
        DELETE FROM authentication
        WHERE registration_number IN (
            SELECT registration_number FROM cards
            WHERE card_status = 'collected'
            AND created_at < datetime('now', '-120 days')
        )
        """
    )
```

## Table Summary

| Table            | Purpose                        | Key Columns                                                                                                                                                                         |
| ---------------- | ------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `students`       | Student records                | `registration_number` (PK), `first_name`, `surname`, `email`, `phone_number`, `programme`, `year_of_study`, `date_of_birth`, `national_id_number`, `faculty`, `registration_status` |
| `cards`          | Track individual ID cards      | `card_id` (PK), `registration_number` (FK), `slot_index`, `card_status`, `batch_id`, `created_at`, `updated_at`                                                                     |
| `authentication` | PIN/OTP credentials & security | `auth_id` (PK), `registration_number` (FK), `otp_hash`, `otp_expiry`, `pin_hash`, **`is_temp_pin`**, `failed_otp_attempts`, `failed_pin_attempts`, `lockout_expiry`                 |
| `audit_log`      | Immutable audit trail          | `log_id` (PK), `timestamp`, `registration_number`, `event_type`, `failure_type`, `session_id`                                                                                       |
| `batches`        | Staff loading sessions         | `batch_id` (PK), `staff_id`, `scan_timestamp`, `total_cards`, `stored_count`, `rejected_count`, `sms_sent_count`                                                                    |

## Schema Initialization

The `init_db.py` script reads `schema.sql` and creates all tables with proper constraints and indices on first run. The database is immutable after creation ŌĆö all modifications happen through application code.
